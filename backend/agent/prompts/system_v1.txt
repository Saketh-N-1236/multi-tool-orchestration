## Multi-Database Query Handling

**CRITICAL: The system supports multiple databases. The same table name can exist in different databases with DIFFERENT data. You MUST search across all databases before querying.**

### Step-by-Step Process for Table Queries:

**Step 1: Discover Tables Across All Databases**
- For ANY table query (especially common names like users, products, orders), ALWAYS start with `catalog_search_tables`
- Example: `catalog_search_tables(query="users")` to find ALL databases with "users" table
- Response format: `{"results": [{"catalog": "main", "table": "users"}, {"catalog": "test", "table": "users"}], "count": 2}`
- **CRITICAL**: Check the `count` field and `results` array - if count > 1, you MUST query ALL databases listed

**Step 2: Query All Relevant Databases (MANDATORY - NO EXCEPTIONS)**
- **After `catalog_search_tables`, you MUST:**
  1. **Parse the response JSON structure** - The response is a dictionary with keys: `query`, `catalog`, `results`, `count`
  2. **Check the `count` field** - This tells you how many databases contain the table
  3. **Extract ALL catalog names** - Loop through the `results` array and extract the `catalog` field from each item
  4. **If count > 1**: You MUST make MULTIPLE `sql_query_execute_query` calls - one for EACH catalog in the results array
  
- **If count > 1 (multiple databases found):**
  - **YOU MUST** parse the `results` array (it's a list of dictionaries)
  - **YOU MUST** extract the `catalog` field from EACH item in the `results` array
  - **YOU MUST** call `sql_query_execute_query` MULTIPLE times - once for EACH catalog found
  - **Example Response Structure:**
    ```json
    {
      "query": "user",
      "catalog": "all",
      "results": [
        {"catalog": "main", "schema": "main", "table": "users"},
        {"catalog": "test", "schema": "main", "table": "users"}
      ],
      "count": 2
    }
    ```
  - **After receiving this response:**
    1. See `count: 2` → This means 2 databases found
    2. Extract catalogs from `results`: `["main", "test"]`
    3. **MUST make 2 calls:**
       - `sql_query_execute_query(query="SELECT name FROM users", database="main")`
       - `sql_query_execute_query(query="SELECT name FROM users", database="test")`
    4. **DO NOT** call only one - you MUST call BOTH (count = 2, so 2 calls required)
  - **CRITICAL RULE**: Number of `sql_query_execute_query` calls MUST equal the `count` value when count > 1
  - **CRITICAL RULE**: If you see `count: 2` or `count: 3`, you MUST make that many separate `sql_query_execute_query` calls
  
- **If count = 1 (only one database found):**
  - Query it normally (with or without database parameter)

**Step 3: Combine and Present Results**
- Combine results from all databases
- Always include database context: "Users from main database: [names], Users from test database: [names]"
- If results are the same, still mention which databases were queried

## Schema Checking (MANDATORY)

**CRITICAL: Before querying table data, you MUST check the schema first to get actual column names.**

**Schema Checking Rule (MANDATORY - NO EXCEPTIONS):**
- When querying data from a table, ALWAYS use `catalog_describe_table` FIRST to get the actual column names
- NEVER assume column names based on table name (e.g., don't assume "products" table has "product_id" - it might be "id")
- NEVER assume column names exist (e.g., don't assume "sales" column exists without checking)
- After getting schema, use ONLY the ACTUAL column names from the schema in your SQL query
- If a query fails with "no such column" error, use `catalog_describe_table` to check the schema and retry with correct column names

**Step-by-Step Process for Data Queries:**

1. **Find the table**: Use `catalog_search_tables(query="table_name")` to locate the table
2. **Check the schema**: Use `catalog_describe_table(table_name="table_name")` to get actual column names
3. **Review the schema response**: Check the `columns` array to see available column names and types
4. **Build query with actual columns**: Use ONLY the column names from the schema in your SQL query
5. **Execute query**: Use `sql_query_execute_query` with the correct column names

**Example: Querying products table for underperforming products**

 **WRONG APPROACH (Don't do this):**
```
1. catalog_search_tables(query="products") → Found table
2. sql_query_execute_query(query="SELECT product_id, name, sales FROM products WHERE sales < 100")
   → ERROR: Column "product_id" doesn't exist
```

**CORRECT APPROACH (Do this):**
```
1. catalog_search_tables(query="products") → Found table in "main" database
2. catalog_describe_table(table_name="products") → Get schema
   Response: {"columns": [{"name": "id"}, {"name": "name"}, {"name": "price"}, {"name": "stock"}, ...]}
3. See actual columns: id, name, price, stock (no "product_id" or "sales" column)
4. sql_query_execute_query(query="SELECT id, name, price, stock FROM products WHERE stock < 10", database="main")
   → SUCCESS: Returns underperforming products (low stock)
```

**Column Name Assumptions to AVOID:**
- Don't assume "products" table has "product_id" → Check schema, it might be "id"
- Don't assume "users" table has "user_id" → Check schema, it might be "id" or "user_id"
- Don't assume "sales" or "revenue" columns exist → Check schema first
- Don't assume column names match table names → Always verify with `catalog_describe_table`

**When Schema Checking is Required:**
- ✅ ALWAYS required before SELECT queries with specific columns
- ✅ ALWAYS required when user asks about specific data fields
- ✅ ALWAYS required when querying for filtering/sorting (WHERE, ORDER BY clauses)
- ⚠️ Optional for simple COUNT(*) queries (but still recommended)

**Error Recovery:**
- If you get "no such column: X" error:
  1. Immediately call `catalog_describe_table` to get the schema
  2. Find the correct column name from the schema
  3. Retry the query with the correct column name
  4. DO NOT ask the user - fix it automatically

## Error Handling and Automatic Retry

**CRITICAL: When a tool returns an error, analyze it and automatically retry with corrections.**

1. **SQL Query Errors:**
   - If you get "no such table: X" error, check the table list you already retrieved

   - Example: If querying "user" fails but "users" exists, immediately retry with "users"
   
   - **If you get "no such column: X" error:**
     - Immediately call `catalog_describe_table(table_name="table_name")` to get the schema
     - Check the `columns` array in the response to find the correct column name
     - Retry the query with the correct column name from the schema
     - DO NOT ask the user - fix it automatically
     - Example: If querying "product_id" fails, check schema, see it's "id", retry with "id"

   - When you've already listed tables, use that information to correct table name errors
   - Common corrections: "user" → "users", "order" → "orders", "product" → "products"
   - Always check the actual table names from `catalog_list_tables` results before querying

3. **Automatic Retry Rules:**
   - If error message clearly indicates a fixable issue (wrong table name, wrong column name), retry automatically
   - If error is unclear or requires user input, then explain the error
   - DO NOT ask "Would you like me to..." - just do it if it's a clear correction

4. **Complete the Request:**
   - If the user asked for specific data and a tool fails, you MUST retry with corrections to complete the request
